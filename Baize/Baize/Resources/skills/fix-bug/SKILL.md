# Fix Bug 工作流

当用户要求修复 Bug 时，按以下步骤执行：

## 步骤 1：理解 Bug 描述
- 仔细阅读用户对 Bug 的描述
- 确认 Bug 的复现步骤（如果用户提供了）
- 确认期望行为 vs 实际行为
- 如果描述不够清晰，询问用户补充信息

## 步骤 2：定位 Bug 代码
- 根据用户描述的功能/文件名，使用 `search_content` 搜索相关代码
- 使用 `search_files` 查找相关文件
- 读取相关代码，理解现有逻辑
- 如果需要，执行测试复现 Bug

## 步骤 3：分析根因
- 对比期望行为和实际行为，找出差异点
- 检查以下常见 Bug 来源：
  - 边界条件处理（空数组、空字符串、首尾元素）
  - 类型转换和可选值解包
  - 异步操作的执行顺序和竞态
  - 状态管理（状态未更新、状态不一致）
  - 资源泄漏（未关闭的文件句柄、未取消的 Task）

## 步骤 4：设计修复方案
- 确定最小化修复方案（只改必要的代码）
- 考虑修复是否会引入新的问题
- 如果 Bug 涉及架构问题，说明但先做最小修复
- 准备好修改的具体代码

## 步骤 5：应用修复
- 使用 `edit_file` 工具修改代码
- 一次只改一个文件，改完确认
- 如果涉及多文件修改，逐个文件修改并确认

## 步骤 6：验证修复
- 建议用户重新运行或执行测试
- 如果有测试用例，运行测试确认通过
- 如果修复引入了新错误，回退并重新分析

## 注意事项
- 修复前先理解完整逻辑，不要盲目修改
- 保持修复的最小化原则，不做无关重构
- 修复后记录根因，避免同类问题再次出现
