# Code Review 工作流

当用户要求代码审查时，按以下步骤执行：

## 步骤 1：获取改动内容
- 执行 `git diff` 查看未暂存的改动
- 执行 `git diff --cached` 查看已暂存的改动
- 如果没有改动，执行 `git log --oneline -5` 查看最近的提交，审查最近一次提交的改动

## 步骤 2：分析改动
对每个改动的文件，分析以下方面：

### 代码质量
- 是否有明显的 bug 或逻辑错误
- 命名是否清晰且一致
- 是否有重复代码可以抽取
- 函数/方法是否过长，是否需要拆分

### 安全性
- 是否有潜在的安全漏洞（如 SQL 注入、XSS、硬编码密钥）
- 是否有敏感信息泄露风险
- 输入验证是否充分

### 性能
- 是否有明显的性能问题（如不必要的循环、N+1 查询）
- 资源是否正确释放
- 是否有内存泄漏风险

### 可维护性
- 是否有足够的注释和文档
- 代码结构是否清晰
- 是否遵循项目的编码规范

## 步骤 3：输出审查报告

按以下格式输出：

### 问题（必须修复）
- [文件:行号] 问题描述

### 建议（推荐修改）
- [文件:行号] 建议描述

### 风险（需要注意）
- [文件:行号] 风险描述

### 总体评价
- 改动质量评分（1-5）
- 总结性评价
